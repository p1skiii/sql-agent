## 0. 全局约束（必须遵守）

1. **必须严格遵守 `CLAUDE.md` 中的代码规范与工程准则。**
   - 所有新建文件、函数命名、模块拆分与注释风格都要以 `CLAUDE.md` 为最高优先级约束。

## 1. 背景与来源

本项目的长期目标是构建一个“自然语言驱动 SQL 的智能助手”，用于帮助用户在工作场景中对业务数据库进行查询和操作。

**MVP1 的设计与实现强烈参考：**

- 仓库内的 `sqlagent.md` 文档  
  - 该文件是对官方 SQL Agent 教程（LangChain / LangGraph SQL Agent）的本地化保存或摘录。  
  - 一切与 SQL Agent 相关的设计，应优先参考 `sqlagent.md` 中的推荐做法与约束。

> 当前阶段（MVP1），只实现最小可用的 **只读 SQL 问答能力**，不做任何写操作、不做 API、不做前端。

## 3. MVP1 的能力范围（Scope）

MVP1 的目标是实现一个**只读 SQL 问答引擎**，通过 LangChain SQL Agent 在 SQLite 沙盒数据库上执行查询。

### 3.1 数据库与数据约束

1. **数据库类型**
   - 使用本地 SQLite 文件作为沙盒数据库（例如 `sandbox/sandbox.db`）。

2. **表结构与数据内容**
   - 至少设计 1–2 张示例业务表（例如：`students` / `courses` 等）。
   - 字段名与字段值必须使用 **英文**（包括人名、城市名、专业名等），例如：
     - `name = "Alice Wang"`, `city = "Shenzhen"`, `major = "Computer Science"`。
   - 目的是避免中文编码/分词带来的额外复杂度，使自然语言查询完全基于 **英文语料**。

3. **初始化方式**
   - 通过 `schema.sql` + `seed.sql` 生成 `sandbox.db`，具体步骤在后续文档中规定。

### 3.2 自然语言查询约束

1. **输入语言**
   - **自然语言输入统一使用英文**（例如：“Find all students whose name contains 'Wang'”）。
   - 单元测试中的所有查询语句也必须使用英文。

2. **查询能力**
   - 输入：英文自然语言问题（只关注单轮问题）。
   - 中间：通过 LangChain SQL Agent 完成：
     - NL（英文） → SQL（SELECT） → 在 SQLite 执行 → 读取结果。
   - 输出：
     - 一段面向用户的英文自然语言回答（summary / answer）。
     - （可选）一个结构化结果对象（列名 + 前几行数据），以便后续在前端展示。

3. **只读安全限制**
   - MVP1 明确**禁止**任何写操作（UPDATE/INSERT/DELETE/DROP/ALTER/TRUNCATE 等）。
   - 策略：
     - 在 Prompt 层强烈约束：只允许生成 SELECT 语句。
     - 在执行前做一个简单的字符串/模式检测：
       - 不以 `SELECT` 开头的 SQL，一律拒绝执行。
     - 对明显表达“修改/新增/删除”意图的英文问题（例如 “delete”, “remove”, “update”, “insert”），可直接返回提示：
       - “MVP1 only supports read-only SELECT queries.”

---

## 4. 明确不在 MVP1 内的功能（Out of Scope）

为了保持 MVP1 的简单和可交付性，以下能力**明确不实现**：

1. **任何写操作**
   - 不支持 UPDATE / INSERT / DELETE / DROP / ALTER 等 SQL 语句的生成与执行。
   - 这类请求在自然语言层面直接给予“当前仅支持查询”的答复。

2. **意图识别模块**
   - 不做独立的“意图分类”（Query vs Modify 等）。
   - MVP1 中默认所有输入都被当作“查询请求”处理，如果检测到明显修改语义，则直接拒绝。

3. **Web API 层（Flask 等）**
   - MVP1 不提供 HTTP 接口。
   - 所有调用仅限于：
     - Python 函数调用；
     - 命令行交互脚本。

4. **前端 UI**
   - 不实现 Web 界面或其他可视化前端。
   - 后续版本会通过 Flask + 前端模板实现 Chat UI。

5. **工作流可视化 / trace 展示**
   - 不做工作流图、时序图 UI。
   - 如需调试，可在控制台或日志中打印必要信息（例如生成的 SQL）。

6. **长期记忆、用户画像、多轮上下文管理**
   - MVP1 只关注单轮问答，不做多轮对话状态管理。

## 5. 成功标准（MVP1 完成的定义）

当满足下列条件时，视为 MVP1 完成：

### 5.1 功能层

1. **自然语言查询闭环**
   - 存在一个清晰的入口（函数或命令行脚本），满足：
     - 输入：一条英文自然语言问题（字符串）。
     - 流程：内部调用 LangChain SQL Agent，在 `sandbox.db` 上执行 SELECT 查询。
     - 输出：
       - 一段英文自然语言回答（例如总结查询结果）。
       - （可选）查询结果的结构化数据（列名 + 前 N 行）。

2. **只读安全约束生效**
   - 针对明显包含 “delete / remove / update / insert / drop / alter / truncate” 等语义或关键字的问题：
     - 不执行任何 SQL；
     - 返回统一的错误或限制提示（例如 “MVP1 only supports read-only SELECT queries.”）。
   - 对于生成的 SQL：
     - 非 `SELECT` 开头的语句一律不执行，并返回错误/提示信息。

3. **英文语料一致性**
   - 数据库中的样例数据全部使用英文字段值。
   - 单元测试与手动测试的自然语言输入全部为英文。

### 5.2 工程与测试层

1. **环境可重复构建**
   - 使用 `uv` 可以在一台新机器上按文档一步构建出可以运行 MVP1 的环境：
     - 安装依赖；
     - 生成并连接 `sandbox.db`；
     - 成功运行 demo 脚本。

2. **基础单元测试（使用 pytest）**
   - 至少编写少量关键路径单测（使用 `pytest`），例如：
     - 给定固定的英文问题，能够生成并执行正确的 SELECT 查询，并返回非空结果；
     - 对于包含“delete / update”等修改语义的输入，能够被安全拒绝，不触发 SQL 执行。
   - 测试用例数量可以不多，但必须覆盖：
     - 正常查询；
     - 非法/不支持操作的拒绝逻辑。

3. **日志与调试信息**
   - 在开发/调试模式下，可以看到：
     - 原始自然语言问题；
     - 生成的 SQL 文本；
     - 查询影响行数或返回行数。
   - 日志输出不需要非常完善，但必须足够支撑后续排错与扩展。

## 6. 后续扩展的衔接说明（为 MVP2/MVP3 预留）

虽然本规范仅定义了 MVP1，但在设计时需要注意为后续版本留好接口边界：

- 将 “自然语言 → SQL → 执行 → 返回结果” 封装成可复用函数/模块，方便后续：
  - 被 Flask `/api/chat` 调用；
  - 被前端 Chat UI 作为后端服务使用；
  - 在更复杂的工作流中被组合使用。

- 未来版本会考虑：
  - 加入 UPDATE/INSERT/DELETE（带预检查与回滚方案）；
  - 引入 LangGraph 或自定义 workflow trace；
  - 提供 Web API 和前端可视化界面。

在实现 MVP1 时，请保持模块边界清晰、接口干净，为后续扩展留下余地。
